Google Docs looks deceptively simple from the outside.

You open a document, type text, share it with someone, watch their cursor move in real time, see their edits appear instantly, add comments, restore previous versions, and trust that nothing will get lost.

But behind that simple user experience is one of the richest system design problems you can study.

A Google Docs-like system combines many hard distributed systems concepts:

Real-time collaboration, concurrent editing, conflict resolution, WebSockets, permission management, large document storage, version history, operation logs, caching, compression, background jobs, cron scheduling, thundering herd avoidance, and authorization.

This case study breaks down how we can design a collaborative document editor from first principles.

The goal is not to exactly reproduce Google’s internal architecture. The goal is to build a strong interview-ready design for a Google Docs-like application and understand the trade-offs deeply.

---

## 1. Problem Statement

Design a collaborative document editor like Google Docs.

Users should be able to create documents, edit them, share them with other users, collaborate in real time, view version history, restore older versions, comment on text, and receive notifications.

The most important challenge is not simple CRUD. A normal notes app can do CRUD.

The real challenge is this:

**Multiple users should be able to edit the same document at the same time without overwriting each other’s work.**

That one requirement changes the entire design.

---

## 2. Functional Requirements

We start with the expected product requirements.

- **Document CRUD:** Users can create, read, update, and delete documents.
- **Sharing and Permissions:** A document owner can share a document with specific users. The owner can assign roles such as viewer, commenter, editor, or owner. The system may also support link-based sharing, where anyone with the link can read, comment, or edit depending on the link settings.
- **Real-Time Collaborative Editing:** Multiple users can edit the same document simultaneously. Each user should see other users’ changes with low latency.
- **Offline Editing:** A user may lose network connectivity, continue editing locally, and sync later when the connection returns.
- **Version History:** Users should be able to view older versions of a document and restore a previous version.
- **Comments:** Users can add comments on document ranges, reply to comments, resolve comments, and mention other users.
- **Notifications:** Users receive notifications when a document is shared with them, when someone comments, when they are mentioned, or when permissions change.
- **Spell Checking:** This can be client-side, server-side, or a hybrid feature depending on complexity.

---

## 3. Non-Functional Requirements

The non-functional requirements are where the system becomes interesting.

- **Real-Time Responsiveness:** Typing must appear locally instantly, and remote edits should propagate quickly. A good target is less than 100–200 ms for collaborative update propagation in normal conditions.
- **High Availability:** Users should be able to open, edit, and collaborate on documents almost all the time.
- **Durability:** Losing user edits is unacceptable. If the UI says an edit is saved, that edit must be durably persisted.
- **Horizontal Scalability:** The system must scale horizontally across millions of users and documents.
- **Eventual Convergence:** Even if users edit concurrently, disconnect, reconnect, or receive operations in different orders, all replicas should eventually converge to the same document state.
- **Security & Authorization:** Permission checks must be secure. Private documents must not leak through reads, search results, public links, or stale caches.
- **Cost Efficiency:** Storing a full copy of a large document after every keystroke would be extremely expensive.

Below is the complete end-to-end architecture layout:

![Google Docs System Design Architecture](/images/blog/google-docs-system-design.svg)

The client talks to the API Gateway for normal HTTP APIs and to the Collaboration Service using WebSockets for real-time editing.

- **Document Service:** Manages document metadata and document lifecycle.
- **Permission Manager:** Authorizes document actions.
- **Collaboration Service:** Handles live editing sessions, user presence, cursors, edit operations, and conflict resolution.
- **Version History Service:** Exposes old versions and restore functionality.
- **Snapshot Worker:** Periodically compacts operation logs into compressed snapshots.
- **Notification Service:** Sends emails, push notifications, or in-app notifications asynchronously.
- **Search Service:** Indexes document titles and content, while making sure search results are permission-filtered.

## 5. Document Data Model

A naive document schema might look like this:

```sql
CREATE TABLE documents (
  id UUID PRIMARY KEY,
  author_id UUID,
  data TEXT,
  updated_at TIMESTAMP
);
```

This works for a toy editor, but it fails for Google Docs.

Why?

Because a Google Doc is not just one blob of text. It has metadata, rich content, permissions, comments, images, embedded objects, version history, and a stream of collaborative edit operations.

Also, if every keystroke updates the same data column, the document row becomes a hot row. Concurrent writes may overwrite each other. Large documents become expensive to rewrite. Version history becomes expensive if every version stores the full document.

A better design separates document metadata from document content.

Metadata is structured and queryable. It belongs in a relational database or distributed SQL store.

Content is large and mutable. It should be stored as compressed snapshots, chunks, and operation logs.

A better metadata schema:

```sql
CREATE TABLE documents (
  doc_id UUID PRIMARY KEY,
  title VARCHAR(255),
  owner_user_id UUID,
  created_by UUID,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  deleted_at TIMESTAMP NULL,
  current_revision BIGINT,
  latest_snapshot_id UUID,
  latest_snapshot_revision BIGINT,
  permission_version BIGINT,
  status VARCHAR(20)
);
```

The important fields are `current_revision`, `latest_snapshot_id`, and `latest_snapshot_revision`.

The document is not simply stored as “current text.” Instead, the current state is reconstructed from:

$$\text{latest snapshot} + \text{operations after that snapshot}$$

This becomes the foundation for both version history and collaborative editing.

## 6. Storing Large Documents

A document can become large. It may contain text, formatting, tables, images, comments, and embedded objects.

Storing a large document body as one huge RDBMS row is not ideal.

Suppose a document is 50 MB and the user changes one word. If we rewrite the whole document, we create huge write amplification. The database replication log grows. Backups become heavier. Cache efficiency drops. The row becomes a contention point.

The better model is:
- **Metadata DB:** Document metadata, owner, title, current revision, snapshot pointer.
- **Object Storage:** Compressed full snapshots, images, embedded objects.
- **Operation Store:** Append-only edit operations keyed by `doc_id` and `revision`.
- **Cache:** Hot active document state and recent operations.

A snapshot table could look like this:

```sql
CREATE TABLE document_snapshots (
  snapshot_id UUID PRIMARY KEY,
  doc_id UUID,
  revision BIGINT,
  object_uri TEXT,
  content_hash VARCHAR(128),
  compressed_size_bytes BIGINT,
  created_at TIMESTAMP
);
```

The actual snapshot content is stored in object storage such as S3, GCS, Azure Blob, or an internal blob store.

The operation log can look like this:

```sql
CREATE TABLE document_operations (
  doc_id UUID,
  revision BIGINT,
  operation_id UUID,
  author_user_id UUID,
  client_id UUID,
  base_revision BIGINT,
  operation_payload JSONB,
  created_at TIMESTAMP,
  PRIMARY KEY (doc_id, revision)
);
```

A user edit is stored as an operation, not as a full document rewrite.

Example operation:

```json
{
  "type": "insert",
  "position": 15,
  "text": "hello",
  "baseRevision": 42
}
```

Another operation:

```json
{
  "type": "delete",
  "position": 20,
  "length": 5,
  "baseRevision": 43
}
```

This is the key design shift:

**A Google Docs-like system stores an ordered stream of edit operations and periodically compacts them into snapshots.**

## 7. Read Path: Opening a Document

When a user opens a document, the system does more than a simple database read.

The flow is:

1. Client sends `GET /documents/{docId}`
2. Gateway authenticates user token.
3. Document Service loads document metadata.
4. Permission Manager checks `READ` permission.
5. Document Service fetches latest snapshot metadata.
6. Content Service loads compressed snapshot from object storage.
7. Operation Store returns operations after `snapshot_revision`.
8. Server or client applies operations to reconstruct current document.
9. Client receives initial document state.
10. Client opens WebSocket connection for real-time collaboration.

If the document is hot and already active in memory, the system can serve it faster from the Collaboration Service or cache.

For cold documents, reconstruction from snapshot plus recent operations is acceptable.

The number of operations after the latest snapshot must be bounded. Otherwise, opening a document may require replaying too many operations. This is why snapshotting and compaction are essential.

---

## 8. Write Path: Editing a Document

When a user types, we should not send the full document to the server.

Instead, the client creates a small operation.

Flow:

1. User types locally.
2. Client applies the edit immediately for instant UX.
3. Client sends operation over WebSocket.
4. Collaboration Server checks `EDIT` permission.
5. Server compares `operation.baseRevision` with current document revision.
6. If needed, server transforms the operation against committed operations.
7. Server assigns the next revision number.
8. Server appends the operation to durable operation log.
9. Server acknowledges the sender.
10. Server broadcasts the canonical operation to other collaborators.
11. Async snapshotter later compacts operations into a snapshot.

The durability rule is important:

**Do not acknowledge an operation as saved until it is durably persisted.**

If the server broadcasts an operation but crashes before persistence, other clients may see text that later disappears. That violates user trust.

## 9. Version History

Version history is one of the most important design areas.

A naive design stores a full document copy every few seconds or after every edit. That becomes expensive quickly.

Example:

$$\text{Document size} = 50\text{ MB}$$
$$\text{Snapshots per day} = 20$$
$$\text{Retention} = 365\text{ days}$$
$$\text{Storage} = 50\text{ MB} \times 20 \times 365 = 365\text{ GB for one active document}$$

This is not scalable.

The better model is:

$$\text{Append-only operation log} + \text{periodic compressed snapshots} + \text{version pointers}$$

Every accepted edit becomes an operation with a monotonically increasing revision number.

Periodically, a background worker creates a full snapshot at a particular revision.

A user-visible version does not always need to store the full document. It can simply point to a revision.

```sql
CREATE TABLE document_versions (
  version_id UUID PRIMARY KEY,
  doc_id UUID,
  revision BIGINT,
  snapshot_id UUID NULL,
  label VARCHAR(255) NULL,
  created_by UUID NULL,
  created_at TIMESTAMP,
  version_type VARCHAR(50)
);
```

For example:
- Version "Before client review" $\rightarrow$ revision 5800
- Version "Final draft" $\rightarrow$ revision 10400

To reconstruct revision 9000:

1. Find latest snapshot at or before revision 9000.
2. Fetch snapshot from object storage.
3. Fetch operations after snapshot up to revision 9000.
4. Apply operations.
5. Return reconstructed document.

Restoring a version should not delete history.

Instead, if the current revision is 12000 and the user restores revision 9000, the system creates a new revision:

$$\text{revision 12001} = \text{restored content from revision 9000}$$

That keeps history auditable.

## 10. Snapshotting Strategy

Snapshots are created asynchronously.

A basic approach is a cron job that snapshots every document every 30 minutes. But that creates a thundering herd.

If 3000 documents all need snapshots every 30 minutes, a naive cron does:
- `00:00` $\rightarrow$ process 3000 documents
- `00:30` $\rightarrow$ process 3000 documents
- `01:00` $\rightarrow$ process 3000 documents

This creates large traffic spikes against the metadata DB, operation store, object storage, and worker fleet.

The better idea is to spread the work.

If we have 3000 documents and a 30-minute snapshot window, process about 100 documents per minute.

Use stable bucketing:

$$\text{bucket} = \text{hash}(\text{doc\_id}) \pmod{30}$$

Then:
- Minute 0 $\rightarrow$ process bucket 0
- Minute 1 $\rightarrow$ process bucket 1
- Minute 2 $\rightarrow$ process bucket 2
- ...
- Minute 29 $\rightarrow$ process bucket 29
- Minute 30 $\rightarrow$ process bucket 0 again

Each document still gets considered once per 30-minute window, but the load becomes smooth. This pattern converts bursty scheduled work into continuous work.

A better schema includes a precomputed bucket:

```sql
CREATE TABLE documents (
  doc_id UUID PRIMARY KEY,
  current_revision BIGINT,
  latest_snapshot_revision BIGINT,
  snapshot_bucket INT,
  last_snapshot_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

Then the scheduler queries:

```sql
SELECT doc_id
FROM documents
WHERE snapshot_bucket = :current_bucket
  AND current_revision > latest_snapshot_revision
LIMIT 1000;
```

Do not scan all documents every minute. Precompute the bucket and index it.

A production architecture should also use a queue:

```text
Scheduler -> Snapshot Task Queue -> Snapshot Workers -> Object Storage + Metadata DB
```

The queue gives backpressure, retries, rate limiting, worker autoscaling, and dead-letter handling.

Snapshot jobs must be idempotent. A good idempotency key is:

$$\text{doc\_id} + \text{target\_revision}$$

If a worker crashes after uploading the snapshot blob but before updating metadata, retrying the same job should be safe.

Write order should be:

1. Compute `target_revision`.
2. Generate snapshot for that exact revision.
3. Upload snapshot blob.
4. Verify hash.
5. Insert snapshot metadata with unique(`doc_id`, `revision`).
6. Update `latest_snapshot_revision` if `target_revision` is newer.

## 11. Avoiding Thundering Herds in Crons

The snapshot example generalizes to many systems.

Any time a design says "Every day at midnight, process all users" or "Every hour, rebuild all recommendations", you should immediately think about thundering herd risk.

The general problem is:

**How do we take a job of size X that must be completed in time Y, and run it as a constant flow of X/Y operations?**

The solution is:

**Hash candidates into Y buckets and process one bucket at a time.**

For daily jobs:

$$\text{bucket} = \text{hash}(\text{user\_id}) \pmod{1440}$$

Then process one bucket per minute.

For hourly jobs:

$$\text{bucket} = \text{hash}(\text{entity\_id}) \pmod{60}$$

Then process one bucket per minute.

This reduces peak load, improves cost efficiency, and avoids cascading failures.

## 12. Concurrent Editing: Why Locks Are Not Enough

Now we come to the heart of the problem.

Suppose the document contains `ABC`.
- User A wants to add D: `ABC` $\rightarrow$ `ABCD`
- User B wants to delete B: `ABC` $\rightarrow$ `AC`

If both users read the old document and write back full copies, one update can overwrite the other.

Timeline:
- **T1:** User A reads `ABC`
- **T2:** User B reads `ABC`
- **T3:** User A writes `ABCD`
- **T4:** User B writes `AC`

Final state becomes `AC`, and User A’s D is lost. If the write order reverses, User B’s deletion is lost. This is the lost update problem.

The first instinct is to use locks.

With document-level pessimistic locking, User A locks the document, User A edits, and everyone else waits until User A releases the lock. This gives correctness but destroys collaboration.

Fine-grained locks over paragraphs or text ranges seem better, but they are difficult because text positions move continuously. If User A locks characters 100–120 and User B inserts 50 characters before that, the locked range must move to 150–170. Now the locking system itself needs transformation logic. Fine-grained locking also causes stale locks, deadlocks, starvation, lock expiration problems, and poor offline editing support.

Optimistic locking with revision numbers is useful for detecting stale writes:

```sql
UPDATE documents
SET content = :newContent,
    revision = revision + 1
WHERE doc_id = :docId
  AND revision = :baseRevision;
```

But optimistic locking only tells us that the client is stale. It does not automatically merge user intent.

For Google Docs, we need something better. The unit of collaboration should not be the full document state. It should be the edit operation.

## 13. Operational Transform

Operational Transform (OT) is a reconciliation algorithm for collaborative editing.

The idea is:

**When an operation arrives that was created on an older version of the document, transform it against operations that already happened, so that the user’s intent is preserved.**

Example:

Initial document: `ABCD`

Positions:
```text
0 1 2 3
A B C D
```

Two users edit concurrently:
- User A deletes B: `Delete(position = 1, length = 1)`
- User B inserts E before C: `Insert(position = 2, text = "E")`

Both operations were created against the same base document `ABCD`.

If delete happens first:
```text
ABCD
delete B
ACD
```
Now User B’s original insert position was 2. But after deleting B, C moved from position 2 to position 1. If we blindly insert at position 2, we get `ACED`, which is wrong because User B intended to insert before C.

So we transform the insert operation:

$$\text{Insert}(2, \text{"E"}) \rightarrow \text{Insert}(1, \text{"E"})$$

Now applying it gives `AECD`.

If insert happens first:
```text
ABCD
insert E before C
ABECD
delete B
AECD
```
Both paths converge to the same result: `AECD`. This is the goal of OT.

## 14. Centralized OT Architecture

A Google Docs-like system can use centralized OT. The server is the authority. Each document has a canonical operation sequence.

```text
Client A      Client B      Client C
   |             |             |
   +-------------+-------------+
                 |
             WebSocket
                 |
       Collaboration Server
                 |
        Operation Sequencer
                 |
           Operation Log
```

Each document has a current revision:

$$\text{doc\_123 current\_revision} = 500$$

Client A sends:

```json
{
  "docId": "doc_123",
  "baseRevision": 498,
  "op": {
    "type": "insert",
    "position": 10,
    "text": "hello"
  }
}
```

The server is already at revision 500. That means operations 499 and 500 happened after the client’s base revision. The server transforms the incoming operation against operations 499 and 500, assigns revision 501, persists it, and broadcasts it.

All clients eventually apply the same ordered operation stream.

Important invariant:

**For each document, all accepted operations have one canonical order.**

Across many documents, scaling is easy because different documents can be assigned to different collaboration servers. Within one hot document, scaling is harder because all operations need one logical order. So we usually route each active document to a single document leader or collaboration session server.

## 15. Client-Side Collaboration Flow

The client cannot wait for the server before showing typed characters. That would feel laggy.

So the client applies local edits immediately.

Flow:
1. User types.
2. Client applies operation locally.
3. Client sends operation to server.
4. Server transforms and accepts canonical operation.
5. Server acknowledges sender.
6. Server broadcasts operation to other clients.
7. Client reconciles local pending operations with server-accepted operations.

The client may hold:

$$\text{server-confirmed document state} + \text{local unacknowledged operations} + \text{remote operations arriving from other users}$$

When remote operations arrive, the client may need to transform local pending operations. This makes collaborative editing complex, but it gives the instant experience users expect.

## 16. OT vs CRDT

Operational Transform is not the only solution. Another major family is CRDTs: Conflict-Free Replicated Data Types.

- **OT is often server-centric:** The server decides canonical operation order, and clients transform operations against server history.
- **CRDTs are more decentralized:** Each element has a unique identity, replicas can merge changes independently, and all replicas eventually converge.

### OT Trade-Offs:
- **Advantages:** Fits centralized systems well, has lower metadata overhead, and works naturally with a server-side sequencer.
- **Disadvantages:** Transform functions are complex, rich text is hard to handle, and correctness bugs can cause divergence.

### CRDT Trade-Offs:
- **Advantages:** Offline-first merging is natural, decentralized collaboration is easier, and strong eventual convergence is built into the data type.
- **Disadvantages:** Metadata overhead can grow rapidly, garbage collection is hard, rich text structures can become complex, and ordering can become unintuitive if not designed carefully.

For this design, OT is a reasonable choice because we assume a centralized collaboration service and a server-side source of truth.

A good interview answer is:

> “I would use Operational Transform for a centralized Google Docs-like design. If the system required peer-to-peer or offline-first decentralized editing, I would consider CRDTs, but I would call out their metadata and garbage collection trade-offs.”

## 17. Operation Log and Reconnect Flow

The operation log is the source of truth for edits after the latest snapshot.

If a client disconnects at revision 100 and reconnects when the server is at revision 150, the client can ask:

```json
{
  "docId": "doc_123",
  "lastSeenRevision": 100
}
```

The server responds with operations 101–150.
- If the revision gap is small, replay operations.
- If the gap is large, send a fresh snapshot plus recent operations.

```python
if gap <= threshold:
    send_missing_operations()
else:
    send_latest_snapshot_and_subsequent_operations()
```

This same mechanism helps with offline editing. A client that edited offline sends operations with old base revisions. The server transforms them against operations that happened meanwhile.

## 18. Permission Management

Permission management answers the question: *Can this user perform this action on this document?*

- **Actions:** `READ_DOCUMENT`, `EDIT_DOCUMENT`, `COMMENT_DOCUMENT`, `SHARE_DOCUMENT`, `DELETE_DOCUMENT`, `RESTORE_VERSION`, `TRANSFER_OWNERSHIP`, `CHANGE_PERMISSION`
- **Roles:** `OWNER`, `EDITOR`, `COMMENTER`, `VIEWER`

A simple permission model might store a JSON object per document:

```json
{
  "123": "Owner",
  "456": "Read",
  "*": "Read"
}
```

Here `"*": "Read"` means anyone with a public link can read. This is simple, but for a scalable system, a row-per-permission model is cleaner.

```sql
CREATE TABLE document_permissions (
  doc_id UUID,
  principal_type VARCHAR(20), -- USER, GROUP, DOMAIN, PUBLIC_LINK
  principal_id VARCHAR(255),
  role VARCHAR(20),           -- OWNER, EDITOR, COMMENTER, VIEWER
  created_by UUID,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  expires_at TIMESTAMP NULL,
  PRIMARY KEY (doc_id, principal_type, principal_id)
);
```

Link sharing should be modeled explicitly:

```sql
CREATE TABLE document_links (
  link_id UUID PRIMARY KEY,
  doc_id UUID,
  token_hash VARCHAR(255),
  role VARCHAR(20),
  enabled BOOLEAN,
  created_by UUID,
  created_at TIMESTAMP,
  expires_at TIMESTAMP NULL
);
```

The public URL contains a random secret token. The database stores only a hash of that token. This protects users if the permission store leaks.

## 19. Permission Check Flow

For document reads:
1. Client sends `GET /documents/{docId}`.
2. Gateway authenticates user if token exists.
3. Document Service calls Permission Manager.
4. Permission Manager checks whether user can `READ_DOCUMENT`.
5. If allowed, Document Service returns document.
6. Otherwise, return `401 Unauthorized` or `403 Forbidden`.

> [!NOTE]
> `401 Unauthorized` represents that user identity is missing or invalid, whereas `403 Forbidden` indicates the user is authenticated but lacks required permission.

For edits:
1. Client sends edit operation over WebSocket.
2. Collaboration Server checks `EDIT_DOCUMENT` permission.
3. If allowed, operation is transformed and persisted.
4. If denied, operation is rejected.

It is not enough to check permission only when the document is opened. Permissions can change during a live editing session.
- **10:00:** User opens doc as editor.
- **10:05:** Owner removes edit permission.
- **10:06:** User continues typing through existing WebSocket.

The system must revoke or downgrade access quickly.

### Good Design:
- Permission decisions are cached briefly.
- Permission changes increment `permission_version`.
- `PermissionChanged` events are published.
- Collaboration servers invalidate active sessions and caches.

Each document metadata row can contain a `permission_version`. When permissions change, increment it and notify active collaboration servers.

---

## 20. Gateway vs Permission Manager

The Gateway should authenticate and route requests, but it should not contain all permission logic.

- **Gateway responsibilities:** Authentication, token validation, rate limiting, routing, and basic request checks.
- **Permission Manager responsibilities:** Authorization decision, role-to-action mapping, link token validation, group/domain permission evaluation, permission cache management, and audit logging.

Document Service and Collaboration Service must enforce authorization themselves. Do not rely only on the Gateway. Internal jobs, future services, or direct service-to-service calls may bypass the Gateway. Authorization must live at the service boundary where data is actually accessed or modified.

---

## 21. Permission Caching and Consistency

Permission checks are on the hot path. Every read, edit, comment, share, and restore needs authorization. Caching is necessary, but stale permissions are dangerous.

Options:
- **Short TTL cache:** Cache key is `doc_id:user_id:action`
- **Document-level ACL cache:** Cache key is `doc_id -> ACL object`
- **Permission versioning:** When permissions change, update permission store, increment `permission_version`, publish `PermissionChanged` event, evict cache, and refresh active sessions.

For sensitive actions like share, delete, transfer ownership, or restore version, prefer synchronous authorization checks. For frequent edit operations, use cached authorization with version invalidation.

> [!IMPORTANT]
> If permission cannot be verified (e.g., database timeout or cache degradation), fail closed. Deny sensitive operations when authorization is uncertain.

## 22. Comments

Comments are attached to document ranges.

A simple schema:

```sql
CREATE TABLE comments (
  comment_id UUID PRIMARY KEY,
  doc_id UUID,
  author_user_id UUID,
  parent_comment_id UUID NULL,
  anchor_start_position BIGINT,
  anchor_end_position BIGINT,
  anchor_revision BIGINT,
  body TEXT,
  status VARCHAR(20), -- OPEN, RESOLVED, DELETED
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

But comments have a tricky problem. If a comment is attached to characters 100–120 and someone inserts 50 characters before that range, the comment should move to 150–170.

So comment anchors must be transformed as document operations happen. The same OT/CRDT logic used for text editing can update comment anchors.

Alternatively, comments can attach to stable structural nodes rather than raw character offsets, for example:

$$\text{paragraph\_id} + \text{offset\_within\_paragraph}$$

This is much more stable than global character positions.

## 23. Notifications

Notifications should be asynchronous.

Examples:
- Document shared with user
- User mentioned in comment
- Comment replied to
- Permission changed
- Version restored

Do not send emails or push notifications synchronously in the request path.

```text
Document/Comment Service -> Event Bus / Queue -> Notification Service -> Delivery
```

Example event:

```json
{
  "eventType": "DOCUMENT_SHARED",
  "docId": "doc_123",
  "actorUserId": "user_1",
  "targetUserId": "user_2",
  "role": "EDITOR",
  "createdAt": "2026-07-01T10:00:00Z"
}
```

The Notification Service can retry failures, deduplicate events, and respect user notification preferences.

## 24. Search

Search is an important hidden requirement. Users need to find documents by title and content. Use a search index such as Elasticsearch, OpenSearch, Solr, or an internal search platform.

```text
Document updated -> DocumentChanged event -> Indexing worker -> Search index
```

But search has a critical security concern: **Search results must be permission-filtered.**

A user should not see private document titles or snippets in search results.

Two approaches:
1. **Query-Time Filtering:** Index everything and filter results through Permission Manager at query time. This is simpler but may be slower for large result sets.
2. **Index-Time Filtering:** Maintain per-user or per-principal access filters directly inside the search index. This is faster but more complex because permission changes must update the search index.

A practical system can use hybrid filtering:
- Search index returns candidate doc IDs.
- Permission Manager filters candidates.
- Only authorized results are returned.

## 25. Caching Strategy

Caching helps reduce latency and cost.

Use cache for:
- Document metadata
- Document ACLs
- Active document state
- Recent operations
- Latest snapshot pointers
- User group membership
- Presence/cursor state

Hot documents can be held in memory by the Collaboration Service. Redis or a distributed cache can store recently accessed metadata and permissions.

But cache is not the source of truth. If the cache is lost, correctness must remain. The system may become slower, but it should not lose data. On crash, another server should reconstruct from durable storage:

$$\text{latest snapshot} + \text{operation log}$$

---

## 26. Compression and Storage Cost

Snapshots should be compressed before being stored in object storage. Text compresses extremely well. Rich-text JSON or structured document formats also compress well.

Compression reduces:
- Object storage cost
- Network transfer cost
- Cache footprint
- Backup size

But compression consumes CPU. So the trade-off is:
- **More CPU** during snapshotting
- **Lower storage/network cost**

For cold snapshots, stronger compression is acceptable. For hot path operations, avoid heavy compression that increases edit latency. Operation payloads may be batched and compressed asynchronously.

---

## 27. Scaling Strategy

The system scales naturally across documents.

- Partition by `doc_id`.
- $\text{hash}(\text{doc\_id}) \rightarrow \text{document shard}$
- $\text{hash}(\text{doc\_id}) \rightarrow \text{collaboration server}$
- $\text{hash}(\text{doc\_id}) \rightarrow \text{operation log partition}$

For millions of documents, this works well. The hard case is one extremely hot document. For a single document with thousands of collaborators, all edit operations need a single logical order. That creates a bottleneck at the document sequencer.

Optimizations include:
- Batching operations
- Compressing broadcast messages
- Throttling cursor/presence updates
- Separating presence from edit operations
- Section-level partitioning for huge documents
- Using hierarchical document structures
- Fanout via pub/sub

Presence updates should not be treated like durable document edits. Cursor movement can be lossy and throttled, whereas text edits must be durable and ordered.

## 28. Failure Modes

A serious design should discuss what can go wrong:

- **Collaboration Server crashes:** Clients reconnect. A new server reconstructs document state from latest snapshot plus operation log. Clients send their last seen revision and receive missed operations.
- **Operation persistence fails:** Do not acknowledge the edit. Client retries.
- **Client retries the same operation:** Use idempotency keys such as `client_id + client_sequence_number`. The server deduplicates repeated operations.
- **Snapshotting fails:** The document still works because operations are durable. Opening may become slower because more operations must be replayed.
- **Snapshot is corrupted:** Verify content hash. Fall back to an older snapshot plus operations.
- **Permission Manager is down:** Fail closed for sensitive operations. Already-open sessions may be allowed briefly only if there is a recently verified permission and product security policy allows it.
- **Cache is lost:** Rebuild from persistent stores. Latency increases, but correctness remains.
- **OT transform code has a bug:** Clients can diverge. This is catastrophic. Use deterministic replay tests, property-based tests, and document hash verification.
- **Public link token leaks:** Allow owner to disable the link. Store only token hash. Support expiration. Maintain audit logs.
- **Search index is stale:** A recently changed document may not appear immediately, or a permission change may not be reflected instantly. Use permission filtering at query time to avoid privacy leaks.

## 31. End-to-End Example

Let us walk through a realistic flow:

1. **Ayush creates a document.**
   The Document Service writes metadata:
   - `doc_id` = `doc_123`
   - `owner_user_id` = `Ayush`
   - `current_revision` = 0
   - `latest_snapshot_revision` = 0
   An empty snapshot is created in object storage.

2. **Ayush shares the document with Shreya as editor.**
   Permission Manager writes:
   - `doc_123`, `USER:shreya`, `EDITOR`
   It increments `permission_version` and publishes a `PermissionChanged` event.

3. **Shreya opens the document.**
   Document Service asks Permission Manager whether Shreya can read. It allows. The client loads the latest snapshot and operations after it.

4. **Both Ayush and Shreya connect to the Collaboration Service using WebSockets.**

5. **Ayush inserts "Hello".**
   Client sends: `Insert(0, "Hello"), baseRevision = 0`. Server accepts as revision 1, persists, and broadcasts.

6. **Shreya inserts " world".**
   Client sends: `Insert(5, " world"), baseRevision = 1`. Server accepts as revision 2.

7. **Concurrent edits:**
   Suppose both edit concurrently. Ayush deletes a word based on revision 2. Shreya inserts text based on revision 2. If Ayush’s operation reaches the server first, Shreya’s operation is transformed. If Shreya’s operation reaches first, Ayush’s operation is transformed. In both cases, OT ensures all clients converge.

8. **Snapshot Worker:**
   Later, the snapshot worker sees that the document has enough operations since the last snapshot. It creates a snapshot at revision 100, compresses it, uploads it, writes snapshot metadata, and updates `latest_snapshot_revision = 100`.

9. **Version History:**
   Can now reconstruct any revision by using snapshots and operations.

## 32. Interview Summary

If you need to explain the entire system in an interview, say something like this:

> “I would design Google Docs as a real-time collaborative editor where document metadata is stored separately from document content. Metadata such as title, owner, current revision, and snapshot pointer can live in a relational or distributed SQL store. The large document body is represented using compressed snapshots in object storage plus an append-only operation log keyed by document id and revision. Users edit by sending operations over WebSocket, not by rewriting the full document. A collaboration server maintains a canonical operation order per active document and uses Operational Transform to merge concurrent edits. Every accepted operation is durably persisted before acknowledgement and then broadcast to collaborators. Version history is implemented using operation logs and periodic snapshots, with user-visible versions pointing to revision numbers. Permissions are handled by a dedicated Permission Manager with roles like owner, editor, commenter, and viewer, plus support for link sharing. Background snapshot jobs are spread across time buckets using hash(doc_id) to avoid thundering herds. The system uses caching for hot documents and permissions, but persistent stores remain the source of truth.”

That single paragraph covers most of the design.

## 33. Key Takeaways

- **Not a CRUD app:** Google Docs is an operation-based collaborative system.
- **Core Data Model:** `metadata + snapshots + operation log`
- **Core Collaboration Model:** Clients send operations, server orders/transforms operations, everyone converges.
- **Core Versioning Model:** `append-only edit history + periodic compressed snapshots + version pointers`
- **Core Permission Model:** `centralized authorization service + ACL/RBAC + careful caching + invalidation`
- **Thundering Herd Avoidance:** Spread scheduled background work across hash buckets.
- **Durability Rule:** Never acknowledge an edit as saved until it is durably persisted.
- **Scaling Limitation:** Scaling across documents is easy; scaling one extremely hot document is hard.

> [!TIP]
> **One-Line Memory Hook:** Google Docs does not coordinate users by locking the document. It coordinates edit operations by ordering, transforming, persisting, and broadcasting them.
